# Design Explorer V2 — Interactive Parametric Explorer

## Context

V1 generates 8 static design directions and presents them in a review UI for comparison.
It works, but it's a gallery — you look, you pick, you move on.

V2 turns the explorer into a **live design instrument**. Instead of comparing pre-made
snapshots, the designer manipulates parametric axes in real time and watches the design
respond. The exploration becomes a conversation between designer intent and generated output.

**Internal context:** Stripe has protodash for on-the-fly generation. Protodash is fast
but doesn't have exploration depth — it generates one direction at a time. Design Explorer V2
fills the gap: structured parametric breadth with the same real-time feedback loop.

The goal: a designer types a prompt, a dev server spins up, and they're immediately
exploring the design space through interactive controls. Snapshots of promising directions
become the starting points for real implementation.

## Core Components

### 1. Parametric Panel UI

The control surface. Lives alongside the preview in a split-screen layout.

**Axis Controls:**

| Axis | Control Type | Values |
|------|-------------|--------|
| Layout Structure | Dropdown or visual carousel | Single column, Asymmetric grid, Horizontal scroll, Modular, Overlapping, Full-bleed, Split-screen, Dashboard, Masonry, Centered stage |
| Information Hierarchy | Segmented control | Typography-led, Data-led, Image-led, Action-led, Narrative, Spatial |
| Interaction Model | Dropdown | Scroll-reveal, Direct manipulation, Progressive disclosure, Command-driven, Hover-rich |
| Typography System | Font pair picker (visual) | Shows actual font rendering, not just names. Grouped by character: humanist, geometric, neo-grotesque, serif, slab, mono, display, mixed |
| Density & Spacing | Slider | Extreme minimal ← → Data-dense. Continuous, not stepped. |
| Color Strategy | Palette swatches | Click to switch between monochromatic, duotone, earth, primaries, dark-native, light, accent-only. Each shows actual colors. |
| Motion Philosophy | Toggle + dropdown | On/off master toggle. When on: functional, choreographed, physics, cinematic, playful, mechanical |
| Visual Metaphor | Card carousel | Newspaper, Control panel, Gallery, Notebook, Blueprint, Storefront, Terminal, Magazine, Dashboard, Library |

**Additional controls:**
- **Randomize** button — randomizes all axes. Good for breaking out of a rut.
- **Randomize unlocked** — lets you lock specific axes and randomize the rest.
- **Lock/unlock per axis** — click the axis label to lock it during randomization.

**Design of the panel itself:**
The panel must be beautifully designed. It IS the product demo. Use the skill's own
anti-slop rules when building it. Dark theme, clean type (DM Sans + JetBrains Mono is
fine here — it's the tool, not the output), tight spacing, responsive controls.

### 2. Live Preview

An iframe or inline-rendered area showing the current design direction.

**Key requirements:**
- Must be isolated from the panel styles (iframe recommended)
- Should show real content, not lorem ipsum — use the brief context to generate
  realistic placeholder data
- Responsive — panel should have a viewport size toggle (mobile / tablet / desktop)
- Loading state while Claude Code regenerates (subtle, not blocking — show the
  previous state with a dim overlay and small spinner)

### 3. Generation Engine (Claude Code integration)

This is the bridge between axis controls and generated code.

**Strategy: Pre-generate + On-demand**

On startup:
1. Claude Code reads the brief and current axis state
2. Generates the initial component for the default axis values
3. Preview loads immediately

On axis change:
1. Panel sends the new axis state (JSON) to a local endpoint or writes to a state file
2. Claude Code detects the change and regenerates the component
3. Vite HMR hot-reloads the preview

**The generation prompt for each render:**
```
Generate a [COMPONENT_TYPE] with these parametric values:
- Layout: [value]
- Hierarchy: [value]
- Interaction: [value]
- Typography: [value]
- Density: [value]
- Color: [value]
- Motion: [value]
- Metaphor: [value]

Brief context: [user's original prompt]
Anti-slop rules: [loaded from references/anti-slop.md]
Design system constraints: [if any]

Output: Single self-contained React component (or HTML file) that can be
rendered in isolation. Include all styles inline or as a style tag.
No external dependencies beyond what's already in the Vite project.
```

**Performance consideration:**
Full regeneration per axis change will feel slow (~5-15 seconds per Claude Code round trip).
Mitigation strategies:
- Debounce rapid changes (wait 500ms after last interaction before regenerating)
- Pre-generate the 8 "cardinal" directions on startup (one per axis at an extreme value)
  and interpolate/blend for intermediate states where possible
- Show previous render during generation with a subtle "updating..." indicator
- For CSS-only axes (density slider, color palette swap), apply changes client-side
  without full regeneration — use CSS custom properties

**CSS-fast axes vs. full-regen axes:**
| Axis | Can do client-side? | How |
|------|-------------------|-----|
| Density | Yes | CSS custom properties for spacing scale |
| Color | Partially | CSS custom properties for palette. New strategy (mono→duotone) needs regen. |
| Motion | Partially | Toggle animations on/off via CSS class. New motion type needs regen. |
| Typography | Partially | Swap font-family via CSS. Different type system (mono→serif) needs regen. |
| Layout | No | Full regeneration required |
| Hierarchy | No | Full regeneration required |
| Interaction | No | Full regeneration required |
| Metaphor | No | Full regeneration required |

This hybrid approach keeps the experience feeling snappy for "surface" tweaks while
accepting a regen delay for structural changes.

### 4. Snapshot System

When the user finds a direction they like, they hit **Snapshot** (📸).

This:
1. Saves the current axis state as JSON
2. Captures the current rendered HTML as a frozen snapshot
3. Adds a thumbnail to the snapshots bar at the bottom
4. Assigns an auto-generated name based on dominant axes (e.g., "Editorial Serif Dense")
   — user can rename

**Grid View** (📊) switches from the live explorer to the V1-style grid showing
all snapshots side by side. User can:
- Compare snapshots
- Delete ones that aren't working
- Select favorites for development
- Return to the live explorer from any snapshot (loads its axis state back into controls)

### 5. Export & Handoff

When the user is done exploring:

**Export Selections** produces:
```json
{
  "brief": "Dashboard card for MRR",
  "snapshots": [
    {
      "name": "The Broadsheet",
      "axes": {
        "layout": "asymmetric-grid",
        "hierarchy": "typography-led",
        "interaction": "scroll-reveal",
        "typography": "serif-forward",
        "density": 0.3,
        "color": "monochromatic",
        "motion": "stillness",
        "metaphor": "newspaper"
      },
      "code_path": "explorations/01-the-broadsheet/",
      "snapshot_html": "snapshots/01-the-broadsheet.html"
    }
  ],
  "timestamp": "2026-03-08T16:30:00Z"
}
```

Claude Code can then take the selected snapshot(s) and build the full production
implementation, carrying forward the axis values as design intent documentation.

## Integration with Protodash

If protodash already handles on-the-fly generation, Design Explorer V2 can sit upstream:

1. **Explorer phase** — designer uses parametric controls to find the right direction
2. **Handoff to protodash** — selected snapshot's axis state becomes protodash input parameters
3. **Protodash refinement** — fast iteration on the chosen direction with protodash's existing speed

The value Design Explorer adds: the structured divergence phase that protodash skips.
Protodash assumes you know what you want. Explorer helps you figure out what you want.

Consider: could the parametric panel be built as a protodash plugin/extension? If protodash
has an extension API, the exploration UI could live inside it rather than as a separate tool.

## Design System Adaptation

The skill ships with generic anti-slop rules and aesthetic directions. For Stripe:

1. **Fork `references/anti-slop.md`** — replace banned fonts with Stripe's actual type
   system. Add Stripe-specific anti-patterns (whatever internal designs feel "template-y").
2. **Fork `references/aesthetic-directions.md`** — keep the structural directions (Swiss,
   Brutalist, Editorial, etc.) but annotate which ones align with Stripe's voice and which
   are useful for internal tools vs. merchant-facing vs. docs.
3. **Add `references/stripe-system.md`** — document the actual design tokens, component
   library, approved patterns. The generation engine references this to stay within the
   system while still exploring.
4. **Constrain the typography axis** — instead of "any Google Font," the font picker shows
   Stripe's approved type stack with maybe 2-3 exploratory options for early-stage ideation.
5. **Color axis** — lock to Stripe's palette but vary how it's applied (which colors
   dominate, how accent is used, dark vs. light mode).

The axes that should remain fully open even within a design system: layout structure,
information hierarchy, interaction model, density, motion philosophy, visual metaphor.
These are structural choices that a design system intentionally doesn't prescribe.

## Open Questions for the Build

1. **Generation latency** — what's acceptable? 2s feels instant, 5s is fine with a
   loading state, 15s starts to break flow. Test and optimize.
2. **Claude Code communication** — WebSocket server vs. file-watch polling vs. CLI
   invocation? File-watch is simplest (write axes.json, Claude Code watches, writes
   output, Vite HMR picks it up). WebSocket is smoother but more infrastructure.
3. **Scope of each generation** — full page? Single component? Configurable? Start with
   single component (what the brief describes) and expand later.
4. **State persistence** — save exploration sessions so you can pick up where you left
   off? Probably yes — serialize snapshots + axis history to a JSON file.
5. **Multi-user** — relevant for Stripe team use? If multiple designers explore the same
   brief, could you share/merge snapshot collections?
6. **Protodash integration depth** — standalone tool that exports to protodash, or
   embedded within protodash as an exploration mode?

---

## Build Priority

1. **Vite scaffold + parametric panel UI** — the controls, the layout, the dark theme.
   Make it look incredible. This is the demo.
2. **CSS-fast axes** — density slider, color palette swap, font toggle. These give
   instant feedback without any generation latency.
3. **Snapshot system** — capture, grid view, compare, select, export.
4. **Claude Code generation bridge** — file-watch approach first, upgrade to WebSocket
   if latency is a problem.
5. **Full-regen axes** — layout, hierarchy, interaction, metaphor. These trigger Claude
   Code and have a loading state.
6. **Protodash integration** — once the standalone tool works.

Steps 1-3 can be built and demoed without any Claude Code integration. The panel itself
with CSS-fast axes and snapshotting is already a useful design tool. Claude Code integration
(steps 4-5) makes it transformative.
